# The Movie Observer (Наблюдатель Кино)

[![Java Version](https://img.shields.io/badge/Java-25-orange.svg)](https://jdk.java.net/25/)
[![Spring Boot](https://img.shields.io/badge/Spring%20Boot-4.0.0--M1-brightgreen.svg)](https://spring.io/projects/spring-boot)
[![PostgreSQL](https://img.shields.io/badge/PostgreSQL-16-blue.svg)](https://www.postgresql.org/)
[![Testcontainers](https://img.shields.io/badge/Testcontainers-checked-blueviolet.svg)](https://testcontainers.com/)

> Веб-сервис каталога фильмов с авторскими рецензиями, обзорами и гибким разграничением прав доступа.

---

## Ключевые возможности

- **Комплексная предметная область**: База данных включает 8 связанных сущностей (пользователи, роли, фильмы, жанры, актеры/режиссеры, каст, рецензии и статьи).
- **Стилизованный интерфейс (The Guardian)**: Серверный рендеринг Thymeleaf с использованием асимметричной сетки, фирменной палитры и шрифтов, полностью локализованный на русский язык.
- **Безопасность и сессии**: Аутентификация на базе JWT (Spring Security 7.x), упакованных в защищенные Cookies с флагами `HttpOnly`, `Secure` и `SameSite=Strict`.
- **Ролевая модель доступа**:
  - **Гость**: Просмотр каталога, одобренных отзывов и опубликованных статей.
  - **Пользователь** (`ROLE_USER`): Написание рецензий с оценками (1–10), отправляемых на премодерацию.
  - **Критик** (`ROLE_CRITIC`): Публикация рецензий в обход модерации + написание аналитических статей.
  - **Редактор** (`ROLE_EDITOR`): Полное управление каталогом (CRUD фильмов/актеров/жанров) и модерация отзывов.
  - **Администратор** (`ROLE_ADMIN`): Управление пользователями и их ролями.
- **Динамический расчет рейтингов**: Средняя оценка фильма автоматически пересчитывается в базе данных при одобрении или добавлении новых рецензий.
- **Журналирование и диагностика**:
  - Фильтр `RequestLoggingFilter` фиксирует HTTP-метод, URI, IP клиента, время выполнения и статус ответа.
  - Поддержка `Spring Boot Actuator` для мониторинга состояния приложения.
- **Автоматические миграции**: Полная схема базы данных и демо-данные накатываются через `Liquibase` автоматически.

## Требования к окружению

Для сборки и запуска приложения вам понадобятся:
- **Java SDK 25**
- **Docker / Docker Desktop** (для запуска базы данных и выполнения интеграционных тестов)

## Быстрый запуск

### 1. Подготовка базы данных
Запустите контейнер PostgreSQL 16 с помощью Docker Compose:
```bash
docker compose up -d
```
> [!NOTE]
> Контейнер настроен на порт `5432` с базой данных `movie_observer`. Все необходимые таблицы и демонстрационные данные создаются и наполняются автоматически при первом запуске приложения с помощью Liquibase.

### 2. Сборка и запуск приложения
Запустите приложение с помощью Gradle-wrapper:

- **На Linux / macOS:**
  ```bash
  ./gradlew bootRun
  ```
- **На Windows:**
  ```cmd
  gradlew.bat bootRun
  ```

После успешного запуска веб-интерфейс будет доступен по адресу:
👉 **[http://localhost:8090](http://localhost:8090)**

## Навигация по ролям

В зависимости от роли авторизованного пользователя в шапке сайта появляются дополнительные ссылки:

| Роль | Видимые элементы навигации |
| :--- | :--- |
| Гость / `ROLE_USER` | Главная, Каталог фильмов, Войти / Регистрация |
| `ROLE_CRITIC` | + **Написать статью** |
| `ROLE_EDITOR` | + **Написать статью**, **Панель редактора** |
| `ROLE_ADMIN` | + **Написать статью**, **Панель редактора** |

**Панель редактора** (`/editor/dashboard`) открывает доступ к:
- Созданию, редактированию и удалению фильмов
- Добавлению актёров и режиссёров
- Модерации пользовательских отзывов (одобрение / отклонение)
- Управлению статьями

## Назначение ролей пользователям

При регистрации через форму `/register` все новые аккаунты автоматически получают роль `ROLE_USER`. Повысить роль можно двумя способами.

### Способ 1 — SQL-запрос через Docker (рекомендуется)

Пока административный UI управления ролями не реализован, роли назначаются напрямую через базу данных.

**Шаг 1.** Откройте интерактивную оболочку PostgreSQL внутри контейнера:
```bash
docker exec -it movie-observer-postgres psql -U postgres -d movie_observer
```

**Шаг 2.** Найдите `id` нужного пользователя по email:
```sql
SELECT id, email, display_name FROM users WHERE email = 'ваш@email.com';
```

**Шаг 3.** Посмотрите таблицу ролей и выберите нужную:
```sql
SELECT id, name FROM roles;
```

Результат:
```
 id |    name
----+------------
  1 | ROLE_USER
  2 | ROLE_CRITIC
  3 | ROLE_EDITOR
  4 | ROLE_ADMIN
```

**Шаг 4.** Назначьте роль пользователю (замените значения на реальные):
```sql
-- Сначала удалите старую роль (если нужно заменить, а не добавить):
DELETE FROM user_roles WHERE user_id = '<uuid пользователя>';

-- Затем вставьте новую:
INSERT INTO user_roles (user_id, role_id)
VALUES ('<uuid пользователя>', <id роли>);
```

**Пример** — сделать пользователя `test@example.com` редактором:
```sql
-- Получить UUID:
SELECT id FROM users WHERE email = 'test@example.com';
-- Допустим, вернул: a1b2c3d4-...

DELETE FROM user_roles WHERE user_id = 'a1b2c3d4-...';
INSERT INTO user_roles (user_id, role_id) VALUES ('a1b2c3d4-...', 3);
```

**Шаг 5.** Выйдите из psql и **перелогиньтесь** в браузере — JWT-токен обновится и новые права вступят в силу.

> [!IMPORTANT]
> Изменение роли вступает в силу только **после нового входа в систему**, так как роли кодируются внутри JWT-токена, выданного при предыдущей авторизации.

---

### Способ 2 — Редактирование миграции (для seeded-данных)

Если вам нужно постоянно иметь тестового пользователя с определённой ролью, добавьте его прямо в файл миграции Liquibase:
`src/main/resources/db/changelog/migrations/02-insert-seed-data.xml`

> [!WARNING]
> После изменения файла миграции необходимо пересоздать базу данных с нуля:
> ```bash
> docker compose down -v
> docker compose up -d
> ```
> Затем запустите приложение — Liquibase применит обновлённые данные автоматически.

## Запуск тестов и отчеты

Проект покрыт модульными и интеграционными тестами (всего 7 ключевых сценариев E2E, включая проверку авторизации, регистрацию, добавление отзывов, проверку Actuator).

### Выполнение тестов
```bash
./gradlew test
```

> [!IMPORTANT]
> Интеграционные тесты используют **Testcontainers** для поднятия реальной базы данных PostgreSQL в Docker. Убедитесь, что Docker запущен перед стартом тестов.
>
> Если вы используете **Docker Desktop на Windows** по протоколу TCP, для обхода ограничений версии API может потребоваться передать параметр:
> ```bash
> ./gradlew test -Dapi.version=1.43
> ```

### Отчет о покрытии (JaCoCo)
```bash
./gradlew jacocoTestReport
```
Отчет будет доступен в папке: `build/reports/jacoco/test/html/index.html`
